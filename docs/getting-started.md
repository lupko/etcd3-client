# Getting started with etcd3-client

This is a quick tour of the most common things you'll do with this client. It is not a
reference — for that, read the docstrings on the client methods or browse the tests under
`tests/`. This doc just shows the shape of typical usage so you can get going.

The examples are intentionally fragmenty: they assume you already have a connected client
called `etcd`. The connect line is in section 1 and is not repeated.

fully tested by the author.

If you are looking for an ultimate reference, there is none. Instead, I recomment to check:
the etcd API documentation, this client's code and its API and possibly tests in this
repo.

## 1. Install & connect

Wheels are published in [GitHub releases](https://github.com/lupko/etcd3-client/releases):

```shell
pip install https://github.com/lupko/etcd3-client/releases/download/0.103.0/etcd3_client-0.103.0-py3-none-any.whl
```

Plain connect — what you'll use most of the time:

```python
import etcd3

etcd = etcd3.client(host="localhost", port=2379)
```

TLS — pass `ca_cert` for server verification, plus `cert_cert` + `cert_key` if mTLS:

```python
etcd = etcd3.client(
    host="etcd.example.com", port=2379,
    ca_cert="/etc/etcd/ca.crt",
    cert_cert="/etc/etcd/client.crt",
    cert_key="/etc/etcd/client.key",
)
```

Username + password auth:

```python
etcd = etcd3.client(host="localhost", port=2379, user="root", password="hunter2")
```

The client is a context manager and accepts a default `timeout` (seconds). Every call also
takes a `timeout_override` if you need a different value for a single call:

```python
with etcd3.client(timeout=2) as etcd:
    etcd.get("/some/key", timeout_override=10)
```

## 2. Keys & values

Keys and values are `str` or `bytes`. Strings are encoded as UTF-8 on the way in. Reads
always return `bytes`.

If you need the etcd `range_end` for a prefix (used in `get_range`, `delete_range`, txn
compares over ranges, etc.), use the `range_end_for_key` function:

```python
from etcd3 import range_end_for_key

prefix = b"users/"
end = range_end_for_key(prefix)   # all keys with prefix "users/" lie in [prefix, end)
```

## 3. Reading

`get` returns `(value, KVMetadata)` if the key exists, or `(None, None)` if it doesn't.
Use it when "missing" is a normal outcome:

```python
value, meta = etcd.get("/config/feature_x")
if value is None:
    ...  # not set
else:
    print(value, meta.mod_revision)
```

`get_strict` is the same but raises `KeyError` if the key is missing — use it when you
expect the key to be there:

```python
value, meta = etcd.get_strict("/config/feature_x")
```

`get_prefix` yields all key-value pairs whose key starts with a given prefix:

```python
for value, meta in etcd.get_prefix("/users/"):
    print(meta.key, value)
```

`get_range` yields all keys in `[range_start, range_end)`:

```python
for value, meta in etcd.get_range(b"a", b"m"):
    ...
```

`get_prefix_paged` (and `get_range_paged`) is what you reach for when the result set is
large. The standard `get_prefix` materializes the whole gRPC response in memory; the paged
variant streams it in chunks. The result is still a single continuous generator — paging
is internal. Sorting is not supported (results come back in key-ascending order):

```python
for value, meta in etcd.get_prefix_paged("/big/keyspace/", page_size=1000):
    process(value)
```

Common knobs you'll use across all the get methods — keys-only reads, sorting, and reads
pinned to a specific revision:

```python
# only the keys, no values (cheaper to transfer)
keys = [meta.key for _, meta in etcd.get_prefix("/users/", keys_only=True)]

# descending sort
for v, m in etcd.get_prefix("/events/", sort_order="desc"):
    ...

# read at a specific historical revision
for v, m in etcd.get_prefix("/users/", revision=12345):
    ...
```

## 4. Writing

`put` is the unconditional write. Returns a `PutResponse` with a header revision:

```python
resp = etcd.put("/config/feature_x", b"on")
print(resp.header.revision)
```

`put_if_not_exists` is an atomic "create only if absent". Returns `(success, response)`:
the response is `None` on failure.

```python
created, resp = etcd.put_if_not_exists("/locks/job-42", b"owner=worker-1")
if created:
    ...  # we got the lock
```

`replace` is an atomic compare-and-swap on the value. Returns `(success, response)`:

```python
swapped, resp = etcd.replace("/state", b"old", b"new")
```

For values that should expire automatically, attach a lease (see section 9).

HINT: `put_if_not_exists` and `replace` are essentially a convenience methods
which create and execute etcd transactions. You can implement same or more
complex, custom atomic operations with transactions.

---

## 5. Deleting

`delete` removes a single key. Returns `(deleted, response)`:

```python
deleted, resp = etcd.delete("/locks/job-42")
```

`delete_prefix` removes everything starting with a prefix — a one-shot bulk delete:

```python
resp = etcd.delete_prefix("/sessions/")
print(resp.deleted)   # how many keys went away
```

`delete_range` removes everything in `[key, range_end)`:

```python
resp = etcd.delete_range(b"a", b"m")
```

All three accept `prev_kv=True` if you want the deleted key-value pairs back in the
response.

## 6. Watching

`watch_response` returns a `(responses_iterator, cancel)` pair. Each item from the iterator
is a `WatchResponse` containing one or more events for the watched key. Iterate it for as
long as you want to consume events; call `cancel()` to stop. Use `start_revision` if you
want to replay history from a known point:

```python
responses, cancel = etcd.watch_response("/config/feature_x", start_revision=last_seen)

try:
    for resp in responses:
        for event in resp:
            print(event.key, event.value)   # PutEvent or DeleteEvent
        # track the high watermark for resume across reconnects
        last_seen = resp.max_mod_revision
finally:
    cancel()
```

`watch_prefix_response` is the same thing for a prefix:

```python
responses, cancel = etcd.watch_prefix_response("/users/")
for resp in responses:
    for event in resp:
        ...
```

IMPORTANT: when building resilient, resumable watcher, you typically want to remember
the last revision that the watcher successfully consumed so that you can resume from
it in case of failure.

When tracking last revision, either find max yourself as you process the events
OR use `WatchResponse.max_mod_revision`. Do not use `revision` field from
the `etcdrpc.WatchResponse` for this. See for example this [issue](https://github.com/gazette/core/pull/336)
for further confirmation.

There are also `watch` / `watch_prefix` variants which flatten to an iterator of `Event`
directly. This is more ergonomic way to consume events if you do not care about
the `WatchResponse` and the utilities present on it.

## 7. Transactions — `txn()` + `TransactionBuilder`

Etcd transactions are "compare a few things, then run one of two op lists depending on
whether all the compares passed". The builder lets you assemble that fluently and the
response gives you typed accessors for each op's result.

### Example A — the trivial case (mimic put_if_not_exists)

"If this key has no creation revision (i.e. doesn't exist), put it":

```python
from etcd3 import TransactionBuilder

tx = TransactionBuilder.new()
tx.compare("/locks/job-42").create.eq(0)
tx.success.put("/locks/job-42", b"owner=worker-1")

response = etcd.txn(tx)
if response.success:
    put_response = response.as_put(0)   # PutResponse for the success op at index 0
    print("acquired at revision", put_response.header.revision)
else:
    print("already held")
```

### Example B — CAS with a fallback fetch

"Replace the value if it equals what we expect; otherwise just give us back the current
value":

```python
tx = TransactionBuilder.new()
tx.compare("/state").value.eq(b"old")
tx.success.put("/state", b"new")
tx.failure.get("/state")

response = etcd.txn(tx)

if response.success:
    print("swapped at", response.as_put(0).header.revision)
else:
    current = response.as_get(0)        # list of (value, meta)
    print("current value is", current[0][0])
```

### Example C — multiple ops in one branch

You can stack ops. The order in which you add them to `success` (or `failure`) is the
order you read them back by index:

```python
tx = TransactionBuilder.new()
tx.compare("/state").value.eq(b"old")
tx.success.delete("/old-key")
tx.success.put("/new-key", b"new")
tx.success.get("/some-other-key")

response = etcd.txn(tx)
assert response.success

del_resp = response.as_del(0)
put_resp = response.as_put(1)
get_resp = response.as_get(2)

print(del_resp.deleted, put_resp.header.revision, len(get_resp))
```

The compare side has the same fluent shape as `value` above: `version`, `create`, `mod`,
and `lease` are the other targets, each with `.eq(x)`, `.ne(x)`, `.gt(x)`, `.le(x)`. Range
compares (a single condition over all keys in `[key, range_end)`) work too:
`tx.compare(prefix, range_end_for_key(prefix)).create.ne(0)`.

`TransactionBuilder.new()` returns a mutable builder. If you need to build several txns
that share a common base, use `TransactionBuilder.immutable()` — each `.compare`,
`.success.*`, `.failure.*` call returns a fresh builder instead of mutating in place.

Nested transactions (`tx.success.txn(other_builder)` plus `response.as_txn(idx)` to read
them) are supported. Not covered here — if you need them, the test suite has the patterns.

## 8. Transactions — legacy `transaction()` API

The client also exposes the older `transaction()` method (with `client.transactions` /
`client.tx` builders and operator-overloaded compares like `tx.value(k) == b"x"`). It is
kept for backward compatibility with the original
[python-etcd3](https://github.com/kragniz/python-etcd3/) and has the same capabilities as
`txn()` but a less ergonomic response shape. Refer to the original python-etcd3 docs, or
to `tests/test_client_txn.py` in this repo, for usage. New code should prefer `txn()`.

## 9. Leases

A lease is a TTL handle. Attach it to a key on put, and the key disappears when the lease
expires (or is revoked):

```python
lease = etcd.lease(ttl=60)
etcd.put("/sessions/abc", b"...", lease=lease)

# ... later ...
lease.revoke()
# /sessions/abc is now gone
```

Refresh the lease before it expires to keep the keys alive:

```python
lease.refresh()
```

You can also ask the lease for its current state: `lease.remaining_ttl` (seconds left,
roundtrips to etcd) and `lease.keys` (the keys currently attached to it).

## 10. Errors

All client errors derive from `etcd3.Etcd3Exception`:

- `ConnectionFailedError` — gRPC says the server is unavailable.
- `ConnectionTimeoutError` — call exceeded the configured timeout.
- `InternalServerError` — etcd returned a server-side error.
- `PreconditionFailedError` — etcd rejected the request as malformed / not allowed.
- `WatchTimedOut` — a watch creation or `watch_once` call timed out.
- `RevisionCompactedError` — you asked for a revision that has been compacted away.
  The earliest still-available revision is on `.compacted_revision`.
